## 詰まったところ・解決の流れ

### 1. `av-scenechange 0.14.1` のプレリュード解決エラー 288件

**症状**: `npm run dev` で 407/505 crate まで進んだ後、`av-scenechange` のビルドで `cannot find type 'Option'` / `'Vec'` / `'Ok'` / `'Some'` / `'Fn'` が連発して失敗。

**経緯**:
- commit `8a19a21` で `screenshots 0.8` → `xcap 0.0.14 + image 0.25` に依存を差し替えていた
- `image 0.25` のデフォルト features に `avif` が含まれ、推移的に `ravif → rav1e → av-scenechange 0.14.1` まで引いてくる
- ARM64 rustc 1.93.0 では `av-scenechange 0.14.1` のプレリュード解決がうまくいかない（`#![no_implicit_prelude]` 的な何かに起因すると推測、深追いせず）

**逃げ道の検討と失敗**:
- こちら側 Cargo.toml で `image = { default-features = false, features = ["png"] }` を試した → **効かない**。xcap の Cargo.toml が `image = "0.25"`（デフォルト有効）と書いているため Cargo の feature unification で avif 付き版に解決される。こちらから下流のデフォルトを殺す手段はない。

**採用した解決**:
- xcap を諦めて `screenshots 0.8` に戻す
- `screenshots 0.8.10` は内部で `image 0.24.9` を再エクスポートしており、そもそも avif チェーンを持たない
- primary monitor の選択は xcap の `.is_primary()` ほど素直ではないが、`Screen::all().iter().find(|s| s.display_info.is_primary)` で明示すれば同等

**学び**: **Cargo の feature unification はダウン方向に伝播しない**。上流クレートがデフォルト features を有効化していたら、こちらで消せない。ARM64 環境では依存ツリーの重量級クレート（特に AV1/動画系）を引き込むかどうかで初回ビルド可否が決まる。

### 2. Windows ページファイル不足（OS error 1455）

**症状**: `widestring` のリンク時に `failed to map object file: ページングファイルが小さすぎる (os error 1455)`。

**原因**: Surface Laptop 7（ARM64 / 16GB / メモリ非換装）で並列ビルド（CPU コア数ぶん、おそらく 10+ 並列）を走らせると、リンカ段階の一時メモリが仮想メモリを使い切る。

**解決**: `CARGO_BUILD_JOBS=2` + `CARGO_INCREMENTAL=0` で並列度を絞る（メモリの `project_markshot_tauri_pivot.md` に記録済みの既知手順）。

### 3. rustc `STATUS_STACK_BUFFER_OVERRUN (0xc0000409)`

**症状**: `tauri` クレートのコンパイル中に rustc.exe がクラッシュ。

**解決**: 同じコマンドでリトライすると通る。根治不能。

### 4. ⭐ 本日の主役 — 非 async command の UI スレッド self-deadlock

**症状**: アプリ起動 → 「New」ボタン押下 → 以下がすべて同時発生：
- メインウィンドウが消えない（タスクバーが少し暗くなるだけ）
- オーバーレイが表示されない
- ESC が効かない
- フロント側の `invoke()` が戻ってこない（= 画面全体が固まる）

**重要な手がかり（eprintln ログ）**:

```
[capture] start_region_capture invoked
[capture] main hidden
[capture] monitor physical=2304x1536 scale=1.5 logical=1536x1024
[capture] screenshot captured data_url_len=971282 info=2304x1536
[capture] building overlay webview window
← ここで止まる。次の "overlay built" も "ERROR" も出ない
```

スクショ取得までは動く → `WebviewWindowBuilder::build()` の中で完全にハング。

**根本原因**:

Tauri 2 の `#[tauri::command] fn`（非 async）は **main thread（UI イベントループ）で実行される**。その handler 内で以下を呼ぶと self-deadlock する：

- `main.hide()` — Windows の `WM_SHOWWINDOW` を UI メッセージループで処理させる必要がある
- `WebviewWindowBuilder::build()` — 新規 WebView 作成は main thread への dispatch 経由

どちらもメインスレッドが **自分の handler 完了を待っている** ため、処理が永久に進まない。

症状の再現性:
- `main.hide()` が処理されない → メインウィンドウが消えない
- build() が完了しない → overlay が出ない
- handler が戻らない → invoke が返らない → フロント UI も止まる

**修正**（commit `7a3795c`）:

```rust
-fn start_region_capture(app: AppHandle, state: State<'_, AppState>) -> Result<(), String> {
+async fn start_region_capture(app: AppHandle, state: State<'_, AppState>) -> Result<(), String> {
```

`async fn` にすると Tauri は Tokio worker thread で実行してくれる。UI スレッドは自由になり、`hide()` メッセージも `build()` dispatch も正常に処理される。

**なぜ今まで気づかなかったか**:

Phase B.2 コミット（`9ec68b6`）のメッセージに注記が残っている：

> cargo check は ARM64 rustc 1.95 OOM で完走せず静的レビューのみで API 整合を確認

つまり **実機動作未検証のまま Phase B.2 を確定していた**。静的レビューでは main thread vs Tokio worker の違いは出ない。本日が初めての実機テストで、即座に表面化した。

## 学び（コラム候補のタネ）

1. **Tauri 2 の 1 文字ルール**: window を操作する command は必ず `async fn` にする。`fn` のままだと UI スレッドが自分を待ってデッドロック。Phase B.2 実装時に見落とした大きな落とし穴。

2. **Cargo の feature unification は一方通行**: 上流クレートが `default-features = true` で依存を書いていたら、下流でデフォルトを殺す手段はない。特に `image 0.25` の avif は ARM64 rustc との相性が悪く、Electron → Tauri 移行で似た罠に踏む可能性が高い。

3. **ARM64 Surface Laptop 7 ビルドの現実**: メモリ 16GB 固定・ページファイル狭小で並列ビルドすると頻繁に OOM / 1455 で落ちる。`CARGO_BUILD_JOBS=2` がデフォルト運用になる。

4. **「cargo check 未完走のまま commit」の借金は必ず返ってくる**: 静的レビューだけで API 整合を確認しても runtime セマンティクス（thread model / message loop）までは見抜けない。実機未検証のまま進めるときは、コミットメッセージに明記しておくと借金の所在がわかって助かる（今回はそれが役に立った）。
